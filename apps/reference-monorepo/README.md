# chemag Reference Monorepo

A realistic demo monorepo exercising every chemag feature end-to-end.

This repo is the canonical fixture used by benchmarks, screenshots, integration
tests, and marketing demos for the chemag toolkit. It is its **own** pnpm
workspace — independent of the outer `chemag-monorepo` development workspace.

## What's inside

| Path | Language | Purpose |
|------|----------|---------|
| `apps/web/` | TypeScript (Next.js 14) | Realistic admin web app — auth, billing, integrations, settings, support, ... |
| `apps/worker/` | TypeScript (Node + pg-boss) | Background worker processing async jobs |
| `apps/api/` | Python (FastAPI) | HTTP API service backing the web app |
| `packages/contracts/` | TypeScript | Generated OpenAPI types shared across web + worker |
| `packages/ui-kit/` | TypeScript | Shared React components |
| `packages/shared-domain/` | TypeScript | Shared value objects (Money, EmailAddress, ...) |
| `src/compounds/` | TypeScript | Flat directory containing every TS chemag compound (web + worker + shared) |

The TypeScript apps share a single chemag workspace at the repo root
(`workspace.yaml`) whose `paths.compounds` points at `src/compounds/`. The
Python service has its own workspace at `apps/api/workspace.yaml` so the
Python plugin can analyze it without TypeScript getting in the way.

## How chemag is wired in

The repo ships with every chemag artifact pre-installed and committed:

| Artifact | Owner | What it does |
|----------|-------|--------------|
| `CLAUDE.md` | Claude Code | Architecture context for Anthropic's CLI |
| `AGENTS.md` | Codex / OpenAI | Same architecture context for Codex |
| `.cursor/rules/architecture.mdc` | Cursor | Inline architecture rules |
| `.github/copilot-instructions.md` | GitHub Copilot | Repository-wide custom instructions |
| `.aider/CONVENTIONS.md` | Aider | Project conventions injected into every chat |
| `.clinerules` | Cline | Architecture rules for Cline sessions |
| `.claude/settings.json` | Claude Code | Pre-installed `chemag check-edit` hook (block mode) |
| `.husky/pre-commit` | Husky | Runs `chemag check` before every commit |

All of these were generated by:

```bash
chemag emit-rules --tool all
chemag install-hooks --tool all
```

Re-running the same commands is idempotent (no-op rows in the install-hooks
summary table; zero diff against the checked-in rule files).

## Trying it locally

```bash
pnpm install                    # workspace deps
pnpm chemag:check               # validates manifests + bond rules (TS + Python)
pnpm chemag:analyze             # workspace summary
pnpm chemag:graph                # Mermaid diagram on stdout
pnpm chemag:emit-rules           # regenerate rule files (no-op vs checked-in copies)
pnpm --filter web build         # Next.js production build
pnpm --filter worker build      # tsc compile of the worker entry point
cd apps/api && python -m pytest # Python plugin smoke tests
```

## What's missing in v1.0

The original WP-018 spec called for a third language sub-tree — a Go service
under `apps/worker-go/`. **The Go plugin lands in WP-021 (Track 2), which is
post-v1.0**, so for now the worker is implemented in TypeScript using
[`pg-boss`](https://github.com/timgit/pg-boss). The Go service will be added
back as a follow-up after WP-021 ships; see `docs/master-plan/STATUS.md`
in the outer chemag repo for the tracking note.

## Where things live in the chemag sense

Every `src/compounds/<name>/compound.yaml` declares the compound's units and
its public surface (`public.ts`). The `apps/web/`, `apps/worker/`, and
`apps/api/` shells import only from a compound's public surface — not from
deep paths inside the compound. The chemag `cross_compound_imports:
public_only` rule enforces that property at check time.

For the chemistry vocabulary used here (element / molecule / reaction /
interface / adapter / buffer), see `workspace.yaml` and the auto-generated
`CLAUDE.md`.
